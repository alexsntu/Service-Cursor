## [v2] Нейроответы — стратегия

### Когда страница ремонта попадает в нейроответ

AI-поисковики (Яндекс Нейро, Google AI Overviews, ChatGPT Search) генерируют ответы **преимущественно на информационные запросы**.

| Тип запроса | Шанс нейроответа | Примеры |
|---|---|---|
| Транзакционный | Очень низкий | «ремонт iPhone Москва», «замена экрана iPhone 16» |
| Коммерческий с условием | Низкий | «гарантия на ремонт iPhone», «сколько стоит замена аккумулятора» |
| Сравнительный / диагностический | **Средний** | «что делать если iPhone не включается», «менять аккумулятор или нет» |
| Информационный про симптом | **Высокий** | «почему iPhone греется», «что значит если не работает Face ID» |
| Информационный про сервис | **Высокий** | «как проходит диагностика в сервисе», «что входит в гарантию на ремонт» |

**Страницы iRepair цитируются в нейроответах через:**
1. **FAQ-блок** — прямые ответы на вопросы про симптомы и процесс ремонта
2. **`.mm-intro-text`** — самодостаточный абзац с конкретными фактами (срок, гарантия, что входит в диагностику)
3. **`WebPage.description`** в JSON-LD — первый источник для Яндекс Нейро и ChatGPT Search
4. **Блог-статьи** — основной канал для широких информационных запросов (вне scope этого скилла)

**Практическое правило:** оптимизация мета-тегов влияет на нейроответы косвенно, через `WebPage.description`. Основная работа — в контенте страницы (`seo-service-page-builder`).

### Флаг нейроответа

Если пользователь явно упоминает нейроответы, AI Overviews, Яндекс Нейро или хочет «цитироваться» — активировать **информационный маршрут** в Step 1 и Step 2d.

## Приоритет трафика: Google лидирует по заявкам

Реальные данные аналитики iRepair:
- **Google**: **55%** органического трафика · заявок **на 50% больше**, чем с Яндекса
- **Яндекс**: **43%** органического трафика · заявок меньше, но поток стабильный
- **Расчётный вес по заявкам**: Google ≈ **60%**, Яндекс ≈ **40%**

**Принцип:** оптимизируем под Google в первую очередь, Яндекс — обязательно поддерживаем. Обе системы критически важны — нельзя жертвовать одной ради другой.

**Формула эффективного веса (на основе доли заявок):**
```
Эффективный вес = (Google-значимость × 0.60) + (Яндекс-значимость × 0.40)
```

iRepair работает в одной точке (Москва) — гео-каскад по нескольким регионам (как у сетевых магазинов с филиалами) не нужен. Один регион:

| Регион | ID Яндекс Wordstat | Инструмент |
|---|---|---|
| Москва | `213` | `wordstat_bulk(phrases=[...], regions=[213])` для замера кластера; `wordstat_top(seed)` — единственный источник LSI/associations, `wordstat_bulk`/`wordstat_regions` LSI не отдают |

### Step 1 — Разбор запроса и определение сущности страницы

Извлечь из сообщения пользователя:
- **Устройство**: iPhone / MacBook / iPad / Apple Watch + конкретная модель (если задана)
- **Тип услуги**: ремонт / замена детали / конкретная неисправность / симптом

- **Keyword data**: если пользователь предоставил список ключей с частотностью — использовать (тогда Step 2b-Wordstat можно пропустить, см. ниже)
- **[v2] Флаг нейроответа**: упомянуты ли нейроответы, AI Overviews, Яндекс Нейро, «цитируемость» — если да, активировать Step 2d
- **URL страницы**: если предоставлен — активировать Step 2f (URL Analysis)

**Маршрут выполнения:**

```
keyword data НЕ предоставлена → Step 2 (SERP, 3 запроса) → [Step 2d если флаг] → Step 2b-Wordstat → Step 2b (fallback) → Step 2c → [Step 2f если есть URL] → Step 3
keyword data предоставлена    → Step 2-mini (2 запроса) → [Step 2d если флаг] → Step 2b (расчёт по данным) → Step 2c → [Step 2f если есть URL] → Step 3

Исключение: пользователь предоставил **измеренную** частоту (не оценку значимости) — тогда её данные приоритетнее, Step 2b-Wordstat можно пропустить.

Если невозможно понять устройство/тип ремонта — задать 1 уточнение.

**Исключение — пагинация / фильтры** (`?page=2`, хэш-параметры): скилл не генерирует мета-теги для этих URL. Решение — `canonical` на первую страницу или `noindex`.

### [v2] Step 2b-Wordstat — Частотный кластерный анализ (приоритетный метод)

**Выполнять всегда**, когда нет измеренной пользователем частоты по ключам (см. правило в Step 1). Использует `wordstat_top`, `wordstat_bulk`, `wordstat_regions`.

**Почему обязателен:** формулировка, которую дал пользователь, или паттерн, который «выглядит правильным» по title конкурентов из Step 2, может не быть самой частотной в кластере. SERP-эвристика (Step 2b ниже) показывает, ЧТО пишут конкуренты в title — но не показывает, ЧТО реально ищут люди в Москве. Выбор ключа для H1/title должен опираться на измерение, а не на предположение или привычную терминологию сайта.

**Алгоритм:**

1. **Seed-запрос** = формулировка из Step 1 (устройство/модель + тип услуги, как их назвал пользователь).
2. **Расширение кластера** — `wordstat_top(seed, num_phrases=30-40)` возвращает `totalCount` seed'а и associations (LSI/related-запросы). К результату добавить вручную обязательные варианты:
   - **латиница ↔ кириллица** для устройства (iPhone↔Айфон, MacBook↔Макбук и т.п.) — тестировать оба написания всегда, победитель непредсказуем
   - тип-слово: «ремонт» ↔ «замена» ↔ «починить» — для симптомных запросов часто выигрывает разговорный глагол («почему не включается»)
   - синонимы деталей: «аккумулятор» ↔ «батарея», «дисплей» ↔ «экран»
   - с гео («Москва») и без — базовая частота нужна, чтобы понять масштаб ниши до наложения гео-фильтра
3. **Частота по Москве** — `wordstat_bulk(phrases=[кластер], regions=[213])`. Общероссийская частота из `wordstat_top` — только справочный ориентир масштаба, не используется для выбора H1/title (федеральный топ вводит в заблуждение для локального сервиса).
4. **Ранжирование** — отсортировать весь кластер по московской частоте (убывание).

**Распределение по тегам:**

| Позиция в ранжированном кластере | Куда идёт |
|---|---|
| Топ-1 (максимальная московская частота, формальный регистр) | Основа H1 и title |
| 2–3 место (LSI-синоним / альтернативная латиница-кириллица) | Уточнение в title / первое предложение description |
| Остальной кластер (разговорные формы, длинный хвост) | Хвост description / FAQ-семена / семантическое ядро (Step 6) |

**[v2] Разделение по интенту (обязательно перед вставкой в тег).** Информационные LSI-фразы («что делать», «почему», «сколько живёт», отзывы, сравнения) — **не вставляются в мета-теги**, даже при высокой частоте: коммерческий тег и информационный интент не смешиваются в одном предложении. Фиксировать их отдельно как FAQ-семена для Step 2c/Step 2d.

**[v2] Покрытие токенов у составных терминов.** Когда в кластере есть словосочетание и его подмножество-подстрока (например «ремонт iPhone 16 Pro» и «ремонт iPhone») — это не всегда конкуренты за одно место. Если более длинный термин содержит все слова короткого, он уже покрывает частотность короткого без потерь: сравнивать их «либо/либо» не нужно. Реальный выбор нужен только когда термины конкурируют за одно и то же тег-место (два разных написания одной сущности — тогда работает антиспам-правило синонимов, см. Step 3) или когда короткий термин не является лемматическим подмножеством длинного.

**[v2] Проверка каннибализации хаб / модель / тип поломки — обязательна, выполняется перед распределением по тегам выше.**

У iRepair три уровня страниц одного устройства, конкурирующих за смежные ключи:
- **Хаб-категория**: «Ремонт iPhone» (`Категории/iPhone/irepair-seo-block.html`)
- **Модель**: «Ремонт iPhone 16 Pro Max»
- **Тип поломки/деталь**: «Замена аккумулятора iPhone»

Если оптимизируемая страница — модель или тип поломки, проверить, не совпадает ли головной термин ранжированного кластера с title/H1 хаб-страницы категории (или наоборот, если оптимизируется хаб). Правило: термин, который уже занят title/H1 существующей страницы того же устройства, исключается из кандидатов на H1/title текущей страницы — берётся следующий по частоте термин кластера, который не занят ни одной из смежных страниц. Занятый термин не выбрасывается из работы совсем — он законно идёт в хвост description как перечисление сопутствующих услуг (это не каннибализация, а нормальная внутренняя перелинковка: «также делаем замену экрана и диагностику»).

Если неизвестно, оптимизирована ли смежная страница под этот термин — спросить пользователя, не предполагать самостоятельно.

**Утечка через паддинг короткого H1.** Если H1 короче 15 симв. и требует расширения (правило в Step 3), не добавлять занятый смежной страницей термин в качестве слова-заполнителя — это возвращает каннибализацию через чёрный ход. Заполнять нейтральными словами из невостребованной части кластера.

**Вывести перед Step 3 таблицу:**

```
## Частотный кластер: [Устройство/Услуга] (Wordstat, регион Москва)

| Запрос | Частота (Москва) | Регистр | Статус |
|---|---|---|---|
| [запрос] | N | формальный / разговорный / латиница / кириллица | 🏆 победитель / LSI / хвост |

→ Запрос пользователя «[исходный seed]» — частота N, место #X в кластере.
→ Рекомендованная основа H1/title: «[топ-запрос]» — частота N ([во сколько раз] выше исходного запроса пользователя).
→ Каннибализация: [не обнаружена / термин «Х» занят страницей «Y», исключён из кандидатов]
```

**Fallback:** если инструменты `wordstat_*` недоступны, заблокированы или вернули пустой результат — перейти к Step 2b (оценка по SERP-частоте ниже) и явно пометить в выводе: «Wordstat недоступен — используется оценка по SERP-частоте, точность ниже измеренной».

### Step 2b — Анализ ключевых слов (SERP-эвристика — fallback, если Wordstat недоступен)

**Когда Wordstat недоступен и keyword data не предоставлена — оценка значимости по SERP:**

Оценить значимость по частоте появления ключа в title/H1 реальных конкурентов (не агрегаторов) из Step 2:

| Частота в топ-10 реальных конкурентов | Оценочная значимость |
|---|---|
| 3+ упоминаний | ~50 — кандидат в title |
| 1–2 упоминания | ~20 — кандидат в description |
| 0 упоминаний | < 15 — пропустить |

Пометить: «значимость оценена по SERP-частоте, не по Wordstat».

**Fallback при малом числе реальных конкурентов (1–2 в топ-10):**
Ключ, присутствующий в title хотя бы одного реального конкурента → кандидат в title (~40). Пометить: «мало реальных конкурентов — оценка по присутствию в title».

---

**Когда keyword data предоставлена внешним источником (не Wordstat-кластером выше) — расчёт эффективного веса:**

> Эта формула — про источник данных (Google vs Яндекс), другая ось, чем московская частота Step 2b-Wordstat. Использовать как fallback, если пользователь принёс готовые цифры значимости без Wordstat-проверки.

```
Эффективный вес = (Google-значимость × 0.60) + (Яндекс-значимость × 0.40)
```

Если источник не разделён по системам — использовать значимость как есть.

**Классификация ключей по порогу:**

| Значимость | Решение |
|---|---|
| ≥ 35 | **Кандидат в title** — точное вхождение даёт реальный прирост трафика |
| 15–34 | **Кандидат в description** — семантический хвост |
| < 15 | Пропустить — не окупает место в теге |

**Fallback когда ни один ключ не достигает порога 35:**
Если максимальный вес < 35 — порог не применять. Использовать ключ с наибольшим весом как основу title, пометить: «ключи ниже порога 35, выбран наиболее весомый».

**Оценка доминирования агрегаторов:**

По каждому запросу подсчитать долю агрегаторов в топ-10:

| Агрегаторов в топ-10 | Статус | Флаг |
|---|---|---|
| 0–1 | Реалистичная цель — iRepair может конкурировать | 🎯 |
| 2–3 | Мониторинг — сложно, но важно отслеживать | 👁 |
| 4+ | Не ставить основной целью — только длинный хвост | — |

Флаг 🎯 / 👁 проставлять в таблице покрытия (Step 4) и семантическом ядре (Step 6).

**Кириллические варианты устройств и деталей:**

| Латиница | Кириллица | Использовать если... |
|---|---|---|
| iPhone | Айфон | топ-1/топ LSI по Wordstat-кластеру, либо эфф. вес ≥ 35 в title / ≥ 15 в description (fallback) |
| MacBook | Макбук | то же правило |
| iPad | Айпад | то же правило |
| Apple Watch | Эпл Вотч | то же правило |
| battery / аккумулятор | батарея | включать оба в description, если оба подтверждены Wordstat или демонд-фильтром |
| дисплей | экран | включать оба, если не превышает лимит |
| замена | ремонт | оба варианта в description |

Кириллика в title — инструмент Яндекс-матчинга. Решение по ней принимается тем же ранжированным кластером Step 2b-Wordstat, что и остальные термины, не отдельным порогом:
- **Основной маршрут (Wordstat-кластер):** топ-1 по московской частоте (включая кириллицу как претендента) → title. Топ LSI победившей фразы → H1.
- **Fallback-маршрут (SERP-эвристика / внешние keyword data):** там действует шкала «Эффективный вес»: ≥ 35 → в title, 15–34 → description, < 15 → не использовать.

### Step 2c — Gap Analysis реальных конкурентов

Выполнять **всегда** (на обоих путях). Двухшаговый процесс.

**Шаг 1 — быстрый анализ SERP-сниппетов:**
Для каждого реального конкурента из топ-10 извлечь прямо из результатов поиска:
- Title (виден в SERP напрямую)
- Наличие meta description
- Кириллика в title/сниппете (да/нет)

**Шаг 2 — WebFetch: максимум 2 параллельных запроса, только если найден ★-конкурент:**
- **1 WebFetch на URL ★-конкурента** (не главная страница — конкретная страница услуги)
- **1 WebFetch на URL iRepair** — для сверки актуального контента и USP

**Fallback при сбое WebFetch:**

### [v2] Step 2f — URL Analysis (опционально, если предоставлен URL)

**Когда выполнять:** пользователь предоставил URL страницы (существующей или планируемой).

**Цель:** проверить, соответствует ли URL-слаг головному ключу кластера из Step 2b-Wordstat, и предложить смену при явном несоответствии.

**Алгоритм:**

1. Извлечь слаг из URL: `zamena-akkumulyatora-iphone-moskva`, `remont-ipad-moskva` и т.п.
2. Определить головной ключ кластера (топ-1 из Step 2b-Wordstat или Step 2b).
3. Оценить покрытие:
   - **Полное** — все значимые слова головного ключа присутствуют в слаге → не менять.
   - **Частичное** — часть слов отсутствует → оценить пользу замены.
   - **Нет покрытия** — слаг технический или не связан с ключом → рекомендовать замену.
4. Проверить слаги URL ★-конкурентов из Step 2 — используют ли они те же ключевые слова.
5. Длина слага: ориентир не более 5–6 слов (устройство/модель/деталь + «moskva»).

**Формат вывода:**

```
## URL Analysis: [текущий URL]

Слаг: [slug]
Головной ключ: [ключ, частота/вес]
Покрытие: полное / частичное / нет
Конкуренты: [домен] использует [slug] — [совпадает / нет]

→ Рекомендация: оставить / заменить на [новый-slug]
→ Причина: [одна фраза]
```

Если рекомендована смена — вывести итоговый URL явно.

### Step 3 — Генерация мета-тегов

> **Правило старшинства:** при противоречии между структурным шаблоном/примером (Step 3a) и именованным правилом (демонд-фильтр USP, антиспам-правило синонимов и т.п.) — побеждает именованное правило. Примеры иллюстрируют формат, не отменяют фильтры.

**Цель не шаблон, а покрытие.** Максимально охватить высоковесовые ключи в рамках технических ограничений.

#### Правила по типу страницы

| Тип страницы | H1 — формат | Гео в H1 | «Ремонт» в title |
|---|---|---|---|
| Устройство (общий ремонт) | `Ремонт [Устройство]` | Нет | Первое слово |
| Конкретная модель | `Ремонт [Устройство] [Модель]` | Нет | Первое слово |
| Тип поломки/детали | `[Замена/Ремонт] [Деталь] [Устройство]` | Нет | Первое слово |
| Сервисная/брендовая | `Сервисный центр Apple в Москве` | **Да — обязательно** | Не ставить |

**H1 — что делает заголовок сильным:**

H1 — навигационный заголовок. Пользователь уже на сайте — коммерческий сигнал не нужен.

**Правила длины:**
- Диапазон: **15–55 симв.** (для ремонтных услуг допустимо до 55 из-за длинных названий моделей)
- Запрещено в H1: «официальный», «авторизованный», «лучший», «дешевый»

**Порядок слов в H1:**
1. Действие (Ремонт / Замена / Диагностика) → устройство → модель
   - ✅ «Ремонт iPhone 16 Pro»
   - ❌ «iPhone 16 Pro ремонт»
2. Для детали/неисправности:
   - ✅ «Замена аккумулятора iPhone»
   - ❌ «Аккумулятор iPhone замена»

#### title — варианты по разным стратегическим осям

**Технические ограничения:**
- Google обрезает после ~65 симв. / 580px
- Яндекс показывает до ~70 симв. без обрезки
- OpenCart: жёсткого лимита символов на meta description в админке не выявлено — ориентир 180–250 симв. без формального потолка (в отличие от CS-Cart-based магазинов с лимитом 255); если он обнаружится на практике — зафиксировать здесь
- Нет keyword stuffing, нет caps, нет `!`, нет цен в ₽
- `iRepair` всегда последним после `|`
- `Москва` обязательно в title

**Три стратегические оси для вариантов:**

| Ось | Что варьировать | Пример |
|---|---|---|
| **А — кириллика** | с/без кириллического варианта | «iPhone» vs «iPhone (Айфон)» |
| **Б — USP-сигнал** | «бесплатная диагностика» / «гарантия» / «срок» | «ремонт с гарантией» vs «ремонт от 30 мин» |
| **В — длина/специфика** | модель + деталь / укороченный Google / полный Яндекс | «Замена экрана iPhone 16» vs «Ремонт iPhone 16 Москва» |

**Процесс генерации title:**
1. Взять ключ с максимальным весом (топ-1 Wordstat-кластера или эфф. вес по fallback) → основа title
2. Построить **2–3 варианта** по разным осям (не только ось А)
3. Для каждого посчитать покрытие ключей и длину
4. Учесть правило первого слова (всегда: «Ремонт» / «Замена» первым)

**Если все варианты > 65 симв. — порядок усечения:**
1. Убрать кириллику в скобках — экономит 8–15 симв.
2. Убрать USP-сигнал, оставить гео
3. Сократить модель до серии (iPhone 16 Pro → iPhone 16)
4. Крайний случай: убрать гео — только услуга + iRepair

**Вывод вариантов — показывать все варианты, не выбирать самостоятельно:**

```
Вариант А: «[title]» ([N] симв.) — [ось: кириллика / USP-сигнал / длина]
→ Покрывает: [что именно]. Сильно для: [Яндекс / Google / оба].

Вариант Б: «[title]» ([N] симв.) — [ось]
→ Покрывает: [что именно]. Сильно для: [Яндекс / Google / оба].

Вариант В: «[title]» ([N] симв.) — [ось] (если есть)
→ Покрывает: [что именно].

→ Рекомендация: Вариант [X] — [одна фраза, обоснование из Step 2b-Wordstat/2b данных].
```

**[v2] Главный принцип композиции (H1/title/description):** содержимое гибких слотов собирается из ранжированной таблицы Step 2b-Wordstat/2c, а не сочиняется по шаблону. Порядок внутри гибкого слота — по убыванию частоты/веса, самое значимое ближе к началу (keyword prominence). Роль-ориентированные слоты (гео-сигнал, USP-якорь, CTA, E-E-A-T) в эту сортировку не входят — их позицию фиксирует AEO-паттерн независимо от частоты. Демонд-фильтр (ниже) и антиспам-правило синонимов определяют, что вообще допустимо включать, — вес решает только порядок среди уже допущенного.

#### meta description

**Технические ограничения:**
- Google Desktop: ~155 симв. видимый сниппет
- Яндекс расширенный сниппет: до 300+ симв.
- Оптимальная длина: **180–250 симв.**

**AEO-паттерн первого предложения (Featured Snippet и AI Overviews):**

Структура: `[Услуга] в iRepair — [прямой ответ на коммерческий запрос]`

AI-системы цитируют предложения, которые:
- Начинаются с объекта запроса (не с «Мы», не с «iRepair»)
- Содержат прямой ответ в первых 10 словах
- Включают гео-сигнал (Москва)
- Заканчиваются конкретным фактом, а не только призывом
- **[v2] Содержат минимум одно конкретное число** («от 30 минут», «180 дней гарантии», «диагностика бесплатно» считается фактом даже без числа — см. JSON-LD раздел ниже для точного требования)

❌ Слабо: «iRepair предлагает ремонт iPhone с гарантией в Москве.»
✅ Сильно: «Ремонт iPhone в iRepair — диагностика бесплатно, гарантия, оригинальные запчасти, Москва.»

**[v2] Демонд-фильтр USP — обязателен перед вставкой любой типовой сервисной фразы:**

Бюджет символов в description ограничен и конкурирует сам с собой: каждый символ на факт без спроса — это символ, не потраченный на LSI-термин из Wordstat-кластера (Step 2b-Wordstat), который реально даёт охват. Перед вставкой USP-фразы («курьер по Москве», «трейд-ин» и т.п.) — проверить её присутствие в Wordstat associations/кластере (composite-запрос «[топ-термин] + USP», например «ремонт iphone курьер»). Частота ~0 → не вставлять как самостоятельный охватный элемент; эта информация уже есть на странице (`.mm-features`) и в контактных разделах сайта.

**Исключение — разрешённые без подтверждения (не более 2 без Wordstat-проверки):**
- «диагностика бесплатно» — ключевой trust-триггер рынка ремонта, включать всегда независимо от Wordstat
- «гарантия [X]» — конверсионный якорь, включать всегда, точный срок — из контента страницы

Остальные USP из списка ниже (курьер, трейд-ин, срок «от N минут») — включать по демонд-фильтру: если фраза встретилась в Wordstat associations или в SERP-заголовках ★-конкурентов (Gap Analysis), считать подтверждённой.

**Структура description — фиксированный порядок:**

```
[AEO-первое предложение: услуга + USP-1 (диагностика/гарантия), USP-2 (по демонд-фильтру) + CTA]    ← до ~155 симв.
[Тип поломок / моделей — по убыванию веса Step 2b-Wordstat/2c]                                       ← приоритет в хвосте
[Кириллические хвосты — если не топ-1/не LSI, но значимость 15–34]                                    ← после перечня
[E-E-A-T: опыт / доверие — только если осталось место]                                               ← в конце хвоста
```

Порядок фиксирован: AI-системы цитируют начало хвоста чаще конца.

- Ключи, не попавшие в title → интегрировать в description естественно
- Оба варианта написания деталей: «аккумулятор» и «батарея», «дисплей» и «экран» — только если оба подтверждены Wordstat/демонд-фильтром, не автоматически
- CTA до 155 симв.: «Запишитесь онлайн» / «Оставьте заявку» / «Вызовите курьера»
- Gap-USP из Step 2c: то, чего не упомянули конкуренты — выдвинуть вперёд

**E-E-A-T в description — приоритет при нехватке места:**

| Приоритет | Сигнал | Фраза | Условие |
|---|---|---|---|
| 1 — обязательно, без демонд-фильтра | Trustworthiness | «диагностика бесплатно», «гарантия [X]» | Всегда, первые 155 симв. |
| 2 — по демонд-фильтру | Expertise | «оригинальные запчасти», «сертифицированные мастера» | Включать, если подтверждено в Wordstat associations или Gap Analysis |
| 3 — при длине > 180, по демонд-фильтру | Experience | «с [год основания] года», «[N] лет на рынке» | В хвосте, только если реального LSI-контента в кластере не осталось |
| 4 — при длине > 220 | Authoritativeness | «авторизованный сервис Apple» (если применимо) | Только если есть место и статус фактически подтверждён |

При нехватке места жертвовать 3 и 4 первыми, затем 2 (если не прошёл демонд-фильтр), но не 1.

**Approved USPs iRepair:**
- «диагностика бесплатно» — Priority 1, без демонд-фильтра
- «гарантия [X] дней/месяцев» (точное значение — из контента страницы) — Priority 1, без демонд-фильтра
- «ремонт от [N] минут» — Priority 2, по демонд-фильтру
- «оригинальные запчасти» — Priority 2, по демонд-фильтру
- «курьер по Москве» / «доставка устройства курьером» — Priority 2, по демонд-фильтру
- «трейд-ин» — Priority 2, по демонд-фильтру
- «запись онлайн» — CTA, не подчиняется демонд-фильтру (это призыв к действию, не охватный ключ)

> Запрещено везде: цены в ₽, эмодзи, `!`
> «официальный» и «авторизованный» — запрещены в H1 и title. В description «авторизованный сервис» допустимо только если это фактически подтверждённый статус iRepair.
> «рассрочка» — использовать только если iRepair реально предлагает рассрочку.

**[v2] Антиспам-правило синонимов:**

Если в одном теге используется несколько ветвей кластера (латиница + кириллица, формальный + разговорный термин) — способ объединения зависит от длины фразы:

- **Однословная калька устройства** («iPhone/Айфон») — скобка при первом упоминании допустима: «iPhone (Айфон)».
- **Многословное название модели** («iPhone 16 Pro/Айфон 16 Про») — скобка сразу после оригинала читается как словарная калька, а не как текст. Вместо этого — естественное упоминание кириллического варианта в отдельном предложении с другой синтаксической ролью (например в CTA: «Записывайтесь на ремонт Айфон 16 Про онлайн»), не рядом с первым упоминанием латиницей.

В обоих случаях — каждое ключевое понятие называется в description один раз, не повторяется отдельными предложениями подряд.

#### [v2] JSON-LD — строгие правила для нейроответов

**`WebPage.description`** — главный источник для Яндекс Нейро и ChatGPT Search.

- Должен содержать **полное первое предложение** meta description (не обрезать на 155 симв. — предложение должно быть завершённым)
- **НЕ начинать** с «iRepair», «Мы», «В нашем сервисе» — начинать с объекта запроса
- **Должен содержать минимум одно конкретное число**: «180 дней гарантии», «от 30 минут», «с [год] года». «Диагностика бесплатно» само по себе числом не считается — если больше нечем закрыть требование, использовать срок гарантии или время ремонта
- Длина: **120–180 симв.**

✅ Правильно: «Ремонт iPhone в iRepair — диагностика бесплатно, гарантия 180 дней, оригинальные запчасти, ремонт от 30 минут.»
❌ Неправильно: «iRepair — сервисный центр Apple в Москве. Здесь вы можете отремонтировать любой iPhone.»

**`LocalBusiness.description`** — 2–3 предложения: специализация + гео + главный USP. Не повторять `WebPage.description` дословно.

**[v2] `LocalBusiness.sameAs` — обязательно:**

```json
"sameAs": [
  "https://yandex.ru/maps/org/irepair/1116056937/"
]
```

Зачем: Яндекс Нейро и ChatGPT Search используют `sameAs` для подтверждения авторитетности организации. Ссылка на верифицированную карточку Яндекс Карт — прямой сигнал доверия.

**Прайс в JSON-LD** (напоминание из CLAUDE.md): только `"priceRange": "&#8381;&#8381;"` в `LocalBusiness` — никаких конкретных сумм ни в `WebPage.description`, ни в `LocalBusiness.description`.
