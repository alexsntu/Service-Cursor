## Что делает пользователь

Пишет запрос, например: страница ремонта iPhone 16 Pro Max, MacBook Pro 14, iPad Air; при необходимости указывает **тип услуги** (замена дисплея, аккумулятора и т.д.).

**Опционально:** дополнительные URL страниц конкурентов сверх обязательного списка — они анализируются вместе с обязательными.

## Шаги скилла (порядок)

### 1. Существующий текст (первый вопрос — всегда)

Перед любым исследованием задать пользователю вопрос:

> **«Есть ли у тебя уже готовый текст для этой страницы? Если да — вставь его, и я использую его в анализе.»**

- **Текст есть** → принять, сохранить как «исходный текст». Использовать его наравне с материалами конкурентов: извлечь структуру, H2/H3, ключевые смыслы, УТП, формулировки — всё, что полезно. Не переписывать дословно, но максимально опираться на логику и содержание.
- **Текста нет / пропустить** → перейти к следующему шагу без задержки.

### 2. Разбор запроса

Устройство и модель, услуга (если есть), черновик slug, учёт приложенных URL конкурентов.

### 3. SERP-исследование

Параллельно ~4 веб-поиска по шаблонам из builder (локальные и федеральные конкуренты, LSI, симптомы/детали). Цель — структура выдачи, темы для H2/H3, зачатки FAQ.

### 3. Страницы конкурентов

**Всегда** анализировать все 5 обязательных конкурентов из раздела «Обязательные конкуренты» выше. Дополнительно — любые URL, которые прислал пользователь.

По каждой странице извлечь: H1/H2/H3, лид-абзац, структуру блоков, FAQ-вопросы, УТП и триггеры доверия. Недостающие темы добавить в ключевую матрицу. Текст страницы — уникальный, не копировать формулировки конкурентов.

### 4. Ключевая матрица (внутренняя)

Кластеры: основной запрос, услуги, симптомы, доверие, гео + ключи с конкурентов + смыслы из исходного текста пользователя (если был). Распределение по H2, лиду, карточкам, FAQ. **Пользователю таблицу не выводить.** Темы для заголовков назначать так, чтобы заранее не закладывать повтор одной и той же формулировки в разных блоках (см. «Уникальность заголовков» ниже).

### 5. Перелинковка (вопрос пользователю)

До генерации HTML: предложить 2–5 логичных внутренних целей и **запросить URL** (или ответ «пропустить»). Без URL — не выдумывать ссылки. Для страниц iPhone в builder есть фиксированный набор ссылок на услуги — см. builder.

### 6. Картинки (вопрос пользователю)

Перед генерацией обязательно спросить: **«Нужны ли картинки для этой страницы?»**

- **Да** → запросить прямые URL изображений и добавить блок фото.
- **Пропустить / не нужны** → блок фото в HTML не добавлять.
- Если пользователь сказал «да», но ссылки не дал → повторно запросить ссылки; без ссылок блок фото не вставлять.

Если блок с фото добавлен:
- на десктопе: 2 изображения в ряд, симметрично, одинакового визуального размера;
- на мобильном: горизонтальный scroll-snap, один кадр по ширине экрана, высота кадра не выше `50vh`;
- под мобильным скроллером: индикатор-полоски (dots), активная полоска обновляется при свайпе.

### 7. Генерация HTML

Строго по правилу `seo--for-service-html-block.mdc`: JSON-LD (`@graph`), встроенный `<style>` (палитра irepair.ru, Liquid Glass, эталонный адаптив из правила), HTML в `.mm-block` целиком без скрытия. Свёртывание («Показать больше») — только если явно используется правило `seo--for-service-html-block--with-collapse.mdc`.

**Содержимое `.mm-block` (логический порядок):** интро (H2 + лид с ответами «кто/где/сроки/диагностика/гарантия»), **полоса из 6 стат-карточек** (не `mm-trust-strip`), сетка услуг **3 или 6** карточек, блок процесса с визуалом, преимущества, **частые неисправности** (сетка симптомов), FAQ (**5–6** вопросов, согласованность с JSON-LD), CTA.

Перед финализацией HTML выполнить **проверку уникальности заголовков** (подробности — раздел ниже): среди всех `<h2>`–`<h4>` в `.mm-block` не должно оставаться пары элементов с одинаковым текстом после нормализации пробелов.

### Уникальность заголовков (обязательно)

**Охват:** любые заголовки уровней H2, H3 и H4 внутри `.mm-block` — заголовки секций, видимый текст вопросов в FAQ, подзаголовки в блоках процесса, симптомов, преимуществ и т.д.

**Нормализация для сравнения:** обрезать пробелы по краям; внутри строки схлопнуть последовательности пробелов и неразрывных пробелов в один обычный пробел. Две строки с разным регистром считаются **разными** (дубликатом считается только полное совпадение канонизированного текста).

**Если найдено совпадение:** переформулировать один из заголовков, сохранив смысл и ключевые слова. Если меняется текст **вопроса FAQ**, синхронно обновить JSON-LD: поле `name` у соответствующего вопроса в `FAQPage` должно **в точности** совпадать с новым видимым заголовком в HTML (как требует правило проекта).

### 8. Самопроверка и выдача

Чеклист из builder (SEO, нейроответы, кириллица в FAQ/симптомах, glass-классы, 6 карточек stat-strip, без цен в ₽, без emoji, телефон/Telegram из констант, **отсутствие дубликатов среди H2–H4** и т.д.). **Телефон в HTML:** видимый текст номера в `<span class="phone_alloka">…</span>` (Alloka); `tel:` и JSON-LD `telephone` — без обёртки, см. правило.

**Выдача:** один финальный HTML в одном кодовом блоке, без комментария до/после.

## Итог одной строкой

Пользователь задаёт тему; скилл **первым делом спрашивает про готовый текст**, исследует выдачу, **всегда анализирует 5 обязательных конкурентов** (+ дополнительные URL от пользователя), уточняет **внутренние ссылки** и **изображения**, собирает страницу под iRepair/OpenCart по правилу проекта и отдаёт готовый HTML.
