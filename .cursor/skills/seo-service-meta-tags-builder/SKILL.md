# SEO Meta Builder — iRepair (Москва)

Генерирует H1, title, meta description и (опционально) OG-теги для страниц **iRepair** — сервисного центра Apple в Москве.

## Приоритет трафика: Google лидирует по заявкам

Реальные данные аналитики iRepair:
- **Google**: **55%** органического трафика · заявок **на 50% больше**, чем с Яндекса
- **Яндекс**: **43%** органического трафика · заявок меньше, но поток стабильный
- **Расчётный вес по заявкам**: Google ≈ **60%**, Яндекс ≈ **40%**

**Принцип:** оптимизируем под Google в первую очередь, Яндекс — обязательно поддерживаем. Обе системы критически важны — нельзя жертвовать одной ради другой.

**Формула эффективного веса (на основе доли заявок):**
```
Эффективный вес = (Google-значимость × 0.60) + (Яндекс-значимость × 0.40)
```

## Workflow (execute in order)

### Step 1 — Разбор запроса и определение сущности страницы

Извлечь из сообщения пользователя:
- **Устройство**: iPhone / MacBook / iPad / Apple Watch + конкретная модель (если задана)
- **Тип услуги**: ремонт / замена детали / конкретная неисправность / симптом
- **Текущие метатеги**: если прислан HTML — извлечь `<title>`, `<meta name="description">`, `<og:*>`
- **Keyword data**: если пользователь предоставил список ключей с частотностью — использовать

**Маршрут выполнения:**

```
keyword data НЕ предоставлена → Step 2 (SERP, 3 запроса) → Step 2b → Step 2c → Step 3
keyword data предоставлена    → Step 2-mini (2 запроса) → Step 2b → Step 2c → Step 3
```

**Step 2c (Gap Analysis) выполняется всегда.**

**Уточняющий вопрос — только если критично отсутствует сущность:**
- Если невозможно понять устройство/тип ремонта — задать 1 уточнение.

**Исключение — пагинация / фильтры** (`?page=2`, хэш-параметры): скилл не генерирует мета-теги для этих URL. Решение — `canonical` на первую страницу или `noindex`.

### Step 2b — Анализ ключевых слов (traffic scoring)

**Когда keyword data не предоставлена — оценка значимости по SERP:**

Оценить значимость по частоте появления ключа в title/H1 реальных конкурентов (не агрегаторов) из Step 2:

| Частота в топ-10 реальных конкурентов | Оценочная значимость |
|---|---|
| 3+ упоминаний | ~50 — кандидат в title |
| 1–2 упоминания | ~20 — кандидат в description |
| 0 упоминаний | < 15 — пропустить |

Пометить: «значимость оценена по SERP-частоте, не по keyword data».

**Fallback при малом числе реальных конкурентов (1–2 в топ-10):**
Ключ, присутствующий в title хотя бы одного реального конкурента → кандидат в title (~40). Пометить: «мало реальных конкурентов — оценка по присутствию в title».

---

**Когда keyword data предоставлена — расчёт эффективного веса:**

```
Эффективный вес = (Google-значимость × 0.60) + (Яндекс-значимость × 0.40)
```

Если источник не разделён по системам — использовать значимость как есть.

**Классификация ключей по порогу:**

| Значимость | Решение |
|---|---|
| ≥ 35 | **Кандидат в title** — точное вхождение даёт реальный прирост трафика |
| 15–34 | **Кандидат в description** — семантический хвост |
| < 15 | Пропустить — не окупает место в теге |

**Fallback когда ни один ключ не достигает порога 35:**
Если максимальный вес < 35 — порог не применять. Использовать ключ с наибольшим весом как основу title, пометить: «ключи ниже порога 35, выбран наиболее весомый».

**Оценка доминирования агрегаторов:**

По каждому запросу подсчитать долю агрегаторов в топ-10:

| Агрегаторов в топ-10 | Статус | Флаг |
|---|---|---|
| 0–1 | Реалистичная цель — iRepair может конкурировать | 🎯 |
| 2–3 | Мониторинг — сложно, но важно отслеживать | 👁 |
| 4+ | Не ставить основной целью — только длинный хвост | — |

Флаг 🎯 / 👁 проставлять в таблице покрытия (Step 4) и семантическом ядре (Step 6).

**Кириллические варианты устройств и деталей (LSI-матрица):**

| Латиница | Кириллица | Использовать если... |
|---|---|---|
| iPhone | Айфон | значимость ≥ 15 в description, ≥ 35 в title |
| MacBook | Макбук | значимость ≥ 15 |
| iPad | Айпад | значимость ≥ 15 |
| Apple Watch | Эпл Вотч | значимость ≥ 15 |
| battery / аккумулятор | батарея | включать оба в description |
| дисплей | экран | включать оба, если не превышает лимит |
| замена | ремонт | оба варианта в description |

Кириллика в title — инструмент Яндекс-матчинга (Яндекс = 40% веса):
- Эффективный вес кириллического варианта ≥ 35 → в title (в скобках: `iPhone (Айфон)`)
- Эффективный вес 15–34 → только description
- Эффективный вес < 15 → не использовать

**Важно:** при расчёте эффективного веса кириллики используй Яндекс-значимость × 0.40 (Google по этому запросу = 0, так как кириллика релевантна только для Яндекса). Поэтому кириллический вариант попадает в title только если его Яндекс-значимость ≥ 87 (что даёт эфф. вес ≥ 35).

### Step 2c — Gap Analysis реальных конкурентов

Выполнять **всегда** (на обоих путях). Двухшаговый процесс.

**Шаг 1 — быстрый анализ SERP-сниппетов:**
Для каждого реального конкурента из топ-10 извлечь прямо из результатов поиска:
- Title (виден в SERP напрямую)
- Наличие meta description
- Кириллика в title/сниппете (да/нет)

**Шаг 2 — WebFetch: максимум 2 параллельных запроса, только если найден ★-конкурент:**
- **1 WebFetch на URL ★-конкурента** (не главная страница — конкретная страница услуги)
- **1 WebFetch на URL iRepair** — для сверки актуального контента и USP

Если ★-конкурентов нет → 1 WebFetch на следующего реального конкурента. WebFetch на iRepair — только при наличии ★-конкурента.

**Fallback при сбое WebFetch:**
Если WebFetch вернул менее 200 символов читаемого текста — пропустить, перейти к следующему. Если все недоступны — использовать SERP-сниппеты как частичный источник; пометить: «данные из сниппета, не с полной страницы».

**Если реальных конкурентов не найдено (все топ-10 — агрегаторы):**
1. Взять страницу наиболее релевантного агрегатора → зафиксировать пробелы
2. Записать: «iRepair может стать первым специализированным сервисом в топ-10 по данному запросу»
3. Вывести рекомендации на основе пробелов

Из каждой страницы извлечь:

| Параметр | Что фиксировать |
|---|---|
| H1 / H2 заголовки | Ключевые слова, наличие гео, формат |
| Title (из `<title>`) | Паттерн, длина, структура |
| FAQ-секция | Есть / нет; если есть — перечислить вопросы |
| Schema.org разметка | Есть / нет, какие типы |
| USP в тексте | Гарантия, диагностика, срок, курьер, трейд-ин, запчасти — что упомянуто |
| Кириллика | Используется / нет |
| Слабые места | Минимум 2–3 конкретных пробела |

**Вывести перед Step 3 блок Gap Analysis:**

```
## Gap Analysis: [Услуга/Устройство]

### Конкурент 1: [domain] ★ / обычный
- Title: «...» ([N] симв.)
- H1: «...»
- FAQ: есть / нет [если есть — вопросы]
- Schema: есть / нет
- USPs: диагностика ✓/✗ · гарантия ✓/✗ · срок ✓/✗ · курьер ✓/✗ · трейд-ин ✓/✗ · оригинальные запчасти ✓/✗
- Кириллика: да / нет
- Слабо: [минимум 2–3 конкретных пробела]

### Конкурент 2: [domain] (если найден)
[аналогично]

### FAQ-семены (для контент-блока):
[список вопросов покупателей]

### Преимущества iRepair — что выдвинуть вперёд:
✅ [преимущество, которого нет у конкурента]
✅ [преимущество, которого нет у конкурента]
⚠️ Добавить в контент: «...» — конкурент закрыл этот интент, iRepair — нет
```

### Step 3 — Генерация мета-тегов

**Цель не шаблон, а покрытие.** Максимально охватить высоковесовые ключи в рамках технических ограничений.

#### Правила по типу страницы

| Тип страницы | H1 — формат | Гео в H1 | «Ремонт» в title |
|---|---|---|---|
| Устройство (общий ремонт) | `Ремонт [Устройство]` | Нет | Первое слово |
| Конкретная модель | `Ремонт [Устройство] [Модель]` | Нет | Первое слово |
| Тип поломки/детали | `[Замена/Ремонт] [Деталь] [Устройство]` | Нет | Первое слово |
| Сервисная/брендовая | `Сервисный центр Apple в Москве` | **Да — обязательно** | Не ставить |

**H1 — что делает заголовок сильным:**

H1 — навигационный заголовок. Пользователь уже на сайте — коммерческий сигнал не нужен.

**Правила длины:**
- Диапазон: **15–55 симв.** (для ремонтных услуг допустимо до 55 из-за длинных названий моделей)
- Запрещено в H1: «официальный», «авторизованный», «лучший», «дешевый»

**Порядок слов в H1:**
1. Действие (Ремонт / Замена / Диагностика) → устройство → модель
   - ✅ «Ремонт iPhone 16 Pro»
   - ❌ «iPhone 16 Pro ремонт»
2. Для детали/неисправности:
   - ✅ «Замена аккумулятора iPhone»
   - ❌ «Аккумулятор iPhone замена»

#### title — варианты по разным стратегическим осям

**Технические ограничения (жёсткие):**
- Google обрезает после ~65 симв. / 580px
- Яндекс показывает до ~70 симв. без обрезки
- Нет keyword stuffing, нет caps, нет `!`, нет цен в ₽
- `iRepair` всегда последним после `|`
- `Москва` обязательно в title

**Три стратегические оси для вариантов:**

| Ось | Что варьировать | Пример |
|---|---|---|
| **А — кириллика** | с/без кириллического варианта | «iPhone» vs «iPhone (Айфон)» |
| **Б — USP-сигнал** | «бесплатная диагностика» / «гарантия» / «срок» | «ремонт с гарантией» vs «ремонт от 30 мин» |
| **В — длина/специфика** | модель + деталь / укороченный Google / полный Яндекс | «Замена экрана iPhone 16» vs «Ремонт iPhone 16 Москва» |

**Процесс генерации title:**
1. Взять ключ с максимальным эффективным весом → основа title
2. Построить **2–3 варианта** по разным осям (не только ось А)
3. Для каждого посчитать покрытие ключей и длину
4. Учесть правило первого слова (всегда: «Ремонт» / «Замена» первым)

**Если все варианты > 65 симв. — порядок усечения:**
1. Убрать кириллику в скобках — экономит 8–15 симв.
2. Убрать USP-сигнал, оставить гео
3. Сократить модель до серии (iPhone 16 Pro → iPhone 16)
4. Крайний случай: убрать гео — только услуга + iRepair

**Вывод вариантов — показывать все варианты, не выбирать самостоятельно:**

```
Вариант А: «[title]» ([N] симв.) — [ось: кириллика / USP-сигнал / длина]
→ Покрывает: [что именно]. Сильно для: [Яндекс / Google / оба].

Вариант Б: «[title]» ([N] симв.) — [ось]
→ Покрывает: [что именно]. Сильно для: [Яндекс / Google / оба].

Вариант В: «[title]» ([N] симв.) — [ось] (если есть)
→ Покрывает: [что именно].

→ Рекомендация: Вариант [X] — [одна фраза, обоснование из Step 2b данных].
```

#### meta description

**Технические ограничения:**
- Google Desktop: ~155 симв. видимый сниппет
- Яндекс расширенный сниппет: до 300+ симв.
- Оптимальная длина: **180–250 симв.**

**AEO-паттерн первого предложения (Featured Snippet и AI Overviews):**

Структура: `[Услуга] в iRepair — [прямой ответ на коммерческий запрос]`

AI-системы цитируют предложения, которые:
- Начинаются с объекта запроса (не с «Мы», не с «iRepair»)
- Содержат прямой ответ в первых 10 словах
- Включают гео-сигнал (Москва)
- Заканчиваются конкретным фактом, а не только призывом

❌ Слабо: «iRepair предлагает ремонт iPhone с гарантией в Москве.»
✅ Сильно: «Ремонт iPhone в iRepair — диагностика бесплатно, гарантия, оригинальные запчасти, Москва.»

**Структура description — фиксированный порядок:**

```
[AEO-первое предложение: услуга + USP-1, USP-2, USP-3 + CTA]    ← до ~155 симв.
[Тип поломок / моделей — перечень работ или устройств]            ← приоритет в хвосте
[Кириллические хвосты — если значимость 15–34]                   ← после перечня
[E-E-A-T: опыт / доверие]                                        ← в конце хвоста
```

Порядок фиксирован: AI-системы цитируют начало хвоста чаще конца.

- Ключи, не попавшие в title → интегрировать в description естественно
- Оба варианта написания деталей: «аккумулятор» и «батарея», «дисплей» и «экран»
- CTA до 155 симв.: «Запишитесь онлайн» / «Оставьте заявку» / «Вызовите курьера»
- Gap-USP из Step 2c: то, чего не упомянули конкуренты — выдвинуть вперёд

**E-E-A-T в description — приоритет при нехватке места:**

| Приоритет | Сигнал | Фраза | Условие |
|---|---|---|---|
| 1 — обязательно | Trustworthiness | «диагностика бесплатно», «гарантия [X]» | Всегда, первые 155 симв. |
| 2 — обязательно | Expertise | «оригинальные запчасти», «сертифицированные мастера» | Всегда |
| 3 — при длине > 180 | Experience | «с [год основания] года», «[N] лет на рынке» | В хвосте |
| 4 — при длине > 220 | Authoritativeness | «авторизованный сервис Apple» (если применимо) | Только если есть место |

При нехватке места жертвовать 3 и 4, но не 1 и 2.

**Approved USPs iRepair:**
- «диагностика бесплатно»
- «гарантия [X] дней/месяцев» (точное значение — из контента страницы)
- «ремонт от [N] минут»
- «оригинальные запчасти»
- «курьер по Москве» / «доставка устройства курьером»
- «трейд-ин»
- «запись онлайн»

> Запрещено везде: цены в ₽, эмодзи, `!`
> «официальный» и «авторизованный» — запрещены в H1 и title. В description «авторизованный сервис» допустимо только если это фактически подтверждённый статус iRepair.
> «рассрочка» — использовать только если iRepair реально предлагает рассрочку.

### Step 3a — Примеры эталонного вывода

Использовать как ориентир. Не копировать дословно.

#### Пример 1: Общий ремонт устройства — «Ремонт iPhone»

```
H1: Ремонт iPhone   (13 симв.)

Вариант А: «Ремонт iPhone в Москве | iRepair»                         (34 симв.) — ось: коротко, Google-first
→ Покрывает: основной запрос + гео. Сильно для: Google.

Вариант Б: «Ремонт iPhone (Айфон) в Москве | iRepair»                 (46 симв.) — ось: кириллика
→ Покрывает: основной + кириллика + гео. Сильно для: Яндекс.
```

### Step 4 — Проверка перед выводом

**Таблица покрытия ключей (с флагами):**

| Ключ | Эфф. вес | Флаг | Где покрыт | Выгода |
|---|---|---|---|---|
| [ключ] | N | 🎯 / 👁 | title / description / не покрыт | Яндекс / Google / — |

**Проверка соответствия требованиям:**

Проверить все параметры внутренне. Если все пройдены — вывести одной строкой: `Все проверки пройдены ✓`. В ответ выводить только строки с нарушением (✗):

| Параметр | Требование |
|---|---|
| H1 длина | 15–55 симв. |
| title длина | ≤ 65 симв. (Google) / ≤ 70 симв. (Яндекс) |
| description длина | 180–250 симв. |
| «Москва» в title | Обязательно |
| «Москва» в description | Обязательно |
| «iRepair» в title | Обязательно (последним после `\|`) |
| Description начинается с услуги/продукта | Не с «iRepair», не с «Мы» |
| AEO-паттерн соблюдён | Прямой ответ в первых 10 словах, есть гео-сигнал |
| Порядок хвоста | Перечень работ → кириллика/LSI → E-E-A-T |
| H1 не дублирует title | H1 не совпадает с основной частью title |
| LSI кириллица/латиница | Оба варианта в description (айфон + iPhone / аккумулятор + батарея) |
| Approved USPs | Только факты из контента страницы — не новые обещания |
| Запрещённые слова | официальный (в title/H1) / эмодзи / ₽ |
| CTA в description | Присутствует до 155 симв. |
| Gap-USP использован | Из Step 2c — приоритет тому, чего нет у конкурента |
| OG-теги | og:title / og:description соответствуют итоговым (могут быть равны или чуть короче) |

> Если ключ с эффективным весом ≥ 35 имеет статус «не покрыт» — вернуться в Step 3 и интегрировать в description. Ключи «не покрыт» с весом < 35 — зафиксировать как ограничение длины, сообщить пользователю.

**Этот шаг обязателен — не пропускать.** После Step 6 (или Step 5, если семантическое ядро не запрашивалось) — всегда задать пользователю вопрос о генерации текста.

**Вывести строго следующее сообщение:**

> Мета-теги готовы. Исследование конкурентов и ключей уже выполнено — повторять поиск не нужно.
>
> Сгенерировать SEO-блок страницы прямо сейчас (intro + FAQ + CTA)?
>
> Если да — ответьте на два вопроса и сразу начнём:
> 1. **Гарантия:** точный срок (например «90 дней», «6 месяцев», «1 год»).
> 2. **Коллапс:** FAQ-блок — свёрнутым (по умолчанию) или развёрнутым сразу?

**Если пользователь отвечает «да» / даёт ответы на вопросы** — немедленно, без дополнительных уточнений, приступить к генерации HTML по правилам `seo-service-page-builder`, используя уже собранные данные из текущего диалога:

- Gap Analysis, USP-приоритеты и FAQ-семена из Step 2c
- Финальные мета-теги из Step 5 (H1, title, description)
- SERP-поиск не повторять — всё уже есть

**Если пользователь отвечает «нет» / «не нужно» / «позже»** — завершить. Сообщить: «Мета-теги сохранены в диалоге — вернитесь к генерации блока в любой момент.»
