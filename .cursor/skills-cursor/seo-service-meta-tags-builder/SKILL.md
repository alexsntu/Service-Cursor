# SEO service meta tags builder (iRepair / Moscow)

Исполняемый скилл: по входной странице (HTML или URL) и (опционально) URL конкурентов делает исследование выдачи в Яндекс и Google, выделяет кластеры частотных релевантных запросов и формирует готовые метатеги по правилу `.cursor/rules/seo--for-service-meta-tags.mdc`.

## Вход (что пользователь может прислать)
- `URL` твоей страницы (или готовый HTML с `<title>` и `<meta name="description">`)
- устройство/модель и тип ремонта (если в запросе есть — использовать, если нет — извлечь из URL/названия/контента)
- `URL` конкурентов (опционально; если есть — анализировать обязательно)

## Порядок работы (execute in order)
### Шаг 1 — Разбор запроса и определение сущности страницы
1. Определить:
   - Устройство (iPhone / MacBook / iPad / Apple Watch + конкретная модель, если задано)
   - Тип услуги (ремонт / замена детали / конкретная неисправность), если указан
   - Текущие метатеги, если пользователь прислал HTML: `<title>`, `meta description`, `og:*` (если присутствует)
2. Уточняющие вопросы (только если критично отсутствует сущность):
   - если невозможно понять устройство/модель/тип ремонта — спросить 1 уточнение у пользователя

### Шаг 2 — Исследование выдачи Яндекс и Google (2 набора SERP)
Выполнить исследования **по обоим поисковикам**, собирая повторяющиеся формулировки из:
- топ выдачи по запросам вида:
  - `[устройство] ремонт Москва`
  - `сервисный центр Apple Москва` (и/или `ремонт Apple Москва`)
  - `[устройство] неисправности/не включается/разбитый экран` (если тип ремонта/симптом известен)
  - `[устройство] диагностика бесплатно Москва` (если это подхватывается конкуренцией в сниппетах)

Фиксировать:
- типичные конструкции в `title` (формат, порядок слов, разделители)
- частотные слова/фразы из сниппетов, особенно: `в Москве`, `сервисный центр`, `ремонт`, названия поломок/деталей
- 6–12 наиболее повторяющихся «кластеров запросов» (на уровне смысла, не копировать дословно чужие метатеги)

### Шаг 2.5 — LSI и варианты написания (кириллица/латиница)
Для каждого ключевого смысла (устройство, деталь, действие, ремонт/диагностика) собрать варианты написания:
- Кириллица: `айфон`, `аккумулятор`, `замена`, `ремонт`, `батарея`
- Латиница: `iPhone`, `battery/accumulator` (если встречается в сниппетах), либо другие латинские написания без транслитных “ошибок”

Далее, при сборке метатегов, обеспечить:
- чтобы в `description` (и при необходимости в `title`) присутствовали и `айфон`, и `iPhone` (либо один из вариантов, если второй не удаётся вместить без переспама)
- чтобы использовались 1–2 LSI-детали: `аккумулятор` и/или `батарея` (не только одно слово)

### Шаг 3 — Анализ конкурентов по URL (если URL присланы)
Если пользователь дал прямые URL конкурентов:
1. Открыть каждую страницу.
2. Извлечь у конкурента:
   - его `<title>`
   - `meta description`
   - (опционально) `og:title` и `og:description`
3. Сопоставить, какие кластеры запросов уже закрываются конкурентов в метатегах.
4. Приоритизировать: что в метатегах конкурентов чаще встречается в топ-результатах.

### Шаг 4 — Сборка «покрытия» ключей для метатегов
Сформировать набор кластеров, который нужно закрыть:
- Основной кластер (главный запрос страницы)
- 1–3 кластера второго уровня (услуга/симптом/деталь)
- 1 кластер доверия (диагностика/гарантия/сроки — только те, которые соответствуют реальному контенту сервиса)
- 1 геокластер (`Москва`)

Правило: включать в метатеги **только то**, что хорошо помещается и читабельно; не «впихивать» все кластеры любой ценой.

### Шаг 5 — Генерация финальных метатегов
Генерировать по нормативу `.cursor/rules/seo--for-service-meta-tags.mdc`:
1. `<title>`: короткий сниппет-формат с `Москва` и `iRepair`, с главным запросом и 1–2 LSI-ключами (если влезает).
2. `<meta name="description">`: <= 250 символов, 1–2 основных ключа + гео + 2–3 факта доверия + CTA.
3. `og:title` и `og:description` (опционально): соответствуют итоговым `title/description`.

### Шаг 6 — Самопроверка и финальный вывод
Перед выводом выполнить чек:
- `Москва` есть в `title` и `description`
- `iRepair` есть в `title` (как минимум)
- `meta description` <= 250 символов (посчитать)
- LSI-кириллица/латиница учтены в `description` (например `айфон` и `iPhone`)
- нет `₽`, нет эмодзи

## Вывод (обязательно)
Вернуть один блок кода с готовыми строками:
```html
<title>...</title>
<meta name="description" content="...">
<!-- optional -->
<meta property="og:title" content="...">
<meta property="og:description" content="...">
```

После кода одной короткой строкой указать:
- `description_chars: N/250`
